## Введение

Данный документ описывает план тестирования веб-платформы **EventQ** - системы поиска и организации мероприятий с функционалом "толпового поиска" (crowd searching). Документ предназначен для команды тестирования, разработчиков и всех участников проекта. Цель тестирования - проверка соответствия реального поведения системы требованиям, описанным в SRS-документе и Use Case диаграмме  

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования выделяются следующие функциональные требования:

* Регистрация и аутентификация пользователей
* Управление профилем пользователя
* Создание и редактирование мероприятий
* Управление видимостью мероприятий (публичные/по заявке/приватные)
* Система поиска и фильтрации мероприятий
* Управление участниками мероприятий (заявки, приглашения, одобрения)
* Система коммуникации (комментарии, чаты, личные сообщения)
* VIP-функционал (приоритет в поиске, закрепление сообщений, кастомизация)
* Административные функции (модерация контента, управление пользователями)
* Система ролей и прав доступа

<a name="quality"/>

## Атрибуты качества

1. **Функциональность:**
   - функциональная полнота: приложение должно выполнять все заявленные функции согласно SRS
   - функциональная корректность: все функции должны работать корректно и безопасно
   - соответствие бизнес-логике: корректное разграничение прав доступа между ролями

2. **Удобство использования:**
   - все функциональные элементы интерфейса имеют понятные названия
   - интерфейс должен соответствовать предоставленным мокапам
   - навигация между разделами не должна превышать 3 клика
   - время обучения базовым функциям - не более 10 минут

3. **Производительность:**
   - время загрузки страницы - не более 2 секунд
   - время отклика на действия пользователя - не более 1 секунды
   - поддержка одновременной работы 10,000 пользователей

4. **Безопасность:**
   - защита от несанкционированного доступа
   - корректное хэширование паролей
   - защита от XSS и CSRF атак
   - валидация пользовательского ввода

5. **Надежность:**
   - доступность системы 99.5%
   - корректная обработка ошибок
   - сохранность данных при сбоях

<a name="risk"/>

## Риски

К рискам проекта можно отнести:

* Ошибки в системе разграничения прав доступа (утечка приватных мероприятий)
* Проблемы с масштабируемостью при большом количестве одновременных пользователей
* Ошибки в бизнес-логике управления участниками (некорректное одобрение/отклонение заявок)
* Уязвимости безопасности (XSS, CSRF, SQL-инъекции)
* Проблемы производительности при работе с чатами в реальном времени
* Несоответствие интерфейса мокапам и ожиданиям пользователей
* Конфликты при одновременном редактировании мероприятий несколькими организаторами
* Потеря сообщений в чатах при сбоях WebSocket-соединения

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения.

### 1. Регистрация и аутентификация

Этот вариант использования необходимо протестировать на:
* успешную регистрацию с валидными данными
* отклонение регистрации с невалидными данными (короткий пароль, занятый логин)
* проверку уникальности имени пользователя
* успешную аутентификацию с корректными данными
* отклонение аутентификации с неверными учетными данными
* восстановление пароля
* выход из системы

### 2. Управление профилем

Этот вариант использования необходимо протестировать на:
* просмотр данных профиля
* редактирование данных профиля
* загрузку и обновление аватара
* настройку уведомлений
* сохранение изменений
* отмену изменений

### 3. Создание и управление мероприятиями

Этот вариант использования необходимо протестировать на:
* создание мероприятия с заполнением всех обязательных полей
* попытку создания мероприятия без обязательных полей
* редактирование созданного мероприятия
* удаление мероприятия
* отмену создания/редактирования мероприятия
* корректное сохранение всех типов данных (текст, даты, категории, теги)

### 4. Управление видимостью мероприятий

Этот вариант использования необходимо протестировать на:
* создание публичного мероприятия (видимость для всех)
* создание мероприятия по заявке (видимость для всех, вступление по одобрению)
* создание приватного мероприятия (видимость только по приглашению)
* проверку корректности отображения мероприятий для разных типов пользователей

### 5. Поиск и фильтрация мероприятий

Этот вариант использования необходимо протестировать на:
* поиск по ключевым словам в названии и описании
* фильтрацию по местоположению
* комбинированную фильтрацию (несколько фильтров одновременно)
* отображение результатов "ничего не найдено"

### 6. Управление участниками мероприятий

Этот вариант использования необходимо протестировать на:
* присоединение к публичному мероприятию
* подачу заявки на участие в мероприятии по заявке
* одобрение заявки организатором
* отклонение заявки организатором
* отправку приглашения на приватное мероприятие
* просмотр списка участников
* удаление участника из мероприятия

### 7. Система коммуникации

Этот вариант использования необходимо протестировать на:
* **Комментарии к мероприятиям:**
  - создание комментария
  - ответ на комментарий (древовидная структура)
  - редактирование своего комментария
  - удаление своего комментария
* **Чат мероприятия:**
  - отправку сообщения в чате
  - получение сообщений (polling)
  - доступ к чату только для участников мероприятия
  - отображение истории чата
* **Личные сообщения:**
  - отправку личного сообщения другому пользователю
  - просмотр истории переписки

### 8. VIP-функционал

Этот вариант использования необходимо протестировать на:
* покупку VIP-статуса
* приоритетное отображение в результатах поиска
* возможность закрепления сообщений в чатах
* применение кастомных тем для мероприятий

### 9. Закрепление сообщений

Этот вариант использования необходимо протестировать на:
* закрепление сообщения VIP-пользователем
* закрепление сообщения организатором
* отображение закрепленного сообщения в верхней части чата

### 10. Административные функции

Этот вариант использования необходимо протестировать на:
* **Модерация контента:**
  - просмотр всех мероприятий (включая приватные)
  - редактирование любого мероприятия
* **Управление пользователями:**
  - просмотр списка всех пользователей
  - блокировку пользователей
* **Системные отчеты:**
  - просмотр аналитики по активности платформы

<a name="nonfunctional"/>

## Нефункциональные требования

Тестирование нефункциональных требований включает:

### Производительность
* Время загрузки страницы не должно превышать 2 секунд
* Время отклика на действия пользователя не должно превышать 1 секунды
* Генерация списка мероприятий с пагинацией должна выполняться за < 1 секунды

### Надежность
* Система должна корректно восстанавливаться после сбоев
* Данные пользователей не должны теряться при сбоях
* Polling для чатов должен работать стабильно

### Совместимость
* Корректная работа в Chrome 90+, Firefox 88+, Safari 14+
* Адаптивный интерфейс для десктопных и мобильных устройств

### Юзабилити
* Интерфейс должен соответствовать предоставленным мокапам
* Все элементы интерфейса должны иметь понятные названия
* Навигация между разделами не должна требовать более 3 кликов

<a name="approach"/>

## Подходы к тестированию

При тестировании будет использован **комбинированный подход**:

### Ручное тестирование
* Функциональное тестирование пользовательского интерфейса
* Тестирование юзабилити и соответствия мокапам
* Тестирование бизнес-логики и сценариев использования

### Регрессионное тестирование
* Повторное выполнение тестов после исправления ошибок
* Проверка критических функций после каждого обновления

<a name="pass"/>

## Представление результатов

Результаты тестирования будут представлены в отдельном документе — **["Результаты тестирования EventQ"](test_result.md)**, включающем:

* Идентификатор тест-кейса
* Описание сценария тестирования
* Действия пользователя
* Ожидаемый результат
* Фактический результат
* Статус выполнения (✅ Пройден / ⚠️ Требуется доработка / ❌ Тест не пройден)
* Приоритет (Критический / Высокий / Средний / Низкий)
* Комментарии и замечания

Формат представления результатов:
* Табличный формат для удобства анализа
* Группировка по функциональным модулям
* Статистика по пройденным/непройденным тестам
* Список критических дефектов
* Рекомендации по устранению проблем

<a name="conclusion"/>

## Выводы

Данный план тестирования охватывает все ключевые функциональные и нефункциональные требования платформы **EventQ**. План включает тестирование:

* Всех пользовательских ролей (Гость, Пользователь, Организатор, VIP-пользователь, Администратор)
* Всех основных функций (регистрация, управление мероприятиями, коммуникация, поиск)
* Критических бизнес-процессов (управление видимостью, система заявок и приглашений)
* Нефункциональных требований (производительность, надежность)

Успешное прохождение всех тестов подтвердит готовность системы к эксплуатации. В случае обнаружения дефектов будет проведен их анализ, определение приоритетов, устранение и повторное тестирование.
